# TON Testnet Wallet

Браузерный self-custodial кошелёк для TON testnet. Без собственного backend — всё работает прямо в браузере.

## Быстрый старт

```bash
npm install

# Настройка переменных окружения (опционально)
cp .env.example .env
# Вставить API-ключ TonCenter в .env (см. ниже)

npm run dev      # http://localhost:5173
npm run build    # production-сборка в dist/
npm test         # юнит-тесты
```

### Переменные окружения

| Переменная | Описание | Обязательная |
|---|---|---|
| `VITE_TONCENTER_API_KEY` | API-ключ TonCenter (testnet) | Нет |

Без ключа кошелёк работает, но TonCenter может возвращать 429 (Too Many Requests) при частых запросах.

Получить бесплатный ключ: [@tonapibot](https://t.me/tonapibot) в Telegram → выбрать Network: **testnet**.

### Тестовые TON

Для получения тестовых токенов: [@testgiver_ton_bot](https://t.me/testgiver_ton_bot) в Telegram — отправь адрес кошелька.

### Запуск тестов

```bash
npm test              # однократный запуск
npm run test:watch    # watch-режим
npm run test:coverage # с покрытием
```

### Сборка для деплоя

```bash
npm run build
npm run preview  # локальный просмотр собранного бандла
```

## Архитектура

### Стек и обоснование выбора

| Инструмент | Роль | Почему выбран |
|---|---|---|
| **React 19** | UI-фреймворк | Стандарт экосистемы, хорошо известен |
| **TypeScript** | Типизация | Строгая типизация критична для финансового приложения: ошибки адресов и сумм должны быть пойманы на этапе компиляции |
| **Zustand 5** | Стейт-менеджмент | Минимальный бойлерплейт; не требует провайдеров; хорошо тестируется без моков фреймворка| 
| **Vite 8** | Бандлер | Быстрый HMR; нативный ESM; нативная поддержка env-переменных |
| **Tailwind CSS** | Стили | Утилитарный CSS без создания отдельных файлов стилей; UI намеренно минималистичен по заданию |
| **@ton/ton + @ton/crypto** | TON SDK | Официальный SDK экосистемы TON; содержит всё необходимое: генерацию мнемоник, деривацию ключей, WalletV4, сериализацию транзакций |
| **Vitest + Testing Library** | Тесты | Нативная интеграция с Vite; совместимый с Jest API |

**Почему не Next.js / Remix?** Приложение полностью клиентское (нет backend по условию задачи). SSR здесь не даёт никакой ценности, но добавляет сложность. Чистый Vite+React — правильный выбор для SPA.

**Почему Zustand, а не Redux?** Redux Toolkit требует action creators, reducers, selectors — избыточный объём кода для небольшого стейта кошелька. Zustand позволяет описать весь стейт в одном файле (~200 строк) с явными мутациями.

### Навигация: state-machine вместо роутера

Приложение не использует react-router. Вместо этого текущий экран (`Screen`) хранится в Zustand-сторе, а `App.tsx` рендерит нужный компонент через `switch`.

**Почему так:** в кошельке навигация жёстко привязана к состоянию (нет кошелька → onboarding; кошелёк заблокирован → unlock; разблокирован → dashboard). URL-based роутинг здесь скорее вреден: пользователь не должен иметь возможность "перейти на `/send`" минуя авторизацию через URL. State-machine делает невалидные переходы невозможными по построению.

**Компромисс:** нельзя использовать кнопку "назад" браузера. Для MVP это приемлемо.

### Хранение данных

| Данные | Где хранится | Обоснование |
|---|---|---|
| Зашифрованная мнемоника | `localStorage` | Нужна между сессиями; защищена AES-256-GCM |
| Адрес кошелька | `localStorage` | Публичные данные, не секрет |
| Приватный ключ (`secretKey`) | Только в памяти | Никогда не пишется в storage; зануляется при блокировке |

## Безопасность

### Шифрование мнемоники

Мнемоника шифруется через `Web Crypto API` (нативный браузерный — без внешних зависимостей):

- **Алгоритм:** AES-256-GCM — обеспечивает и конфиденциальность, и аутентификацию (auth tag проверяет целостность при дешифровании)
- **Деривация ключа:** PBKDF2 с 100 000 итерациями SHA-256 и 128-битной солью — делает перебор паролей дорогостоящим
- **IV и соль:** генерируются случайно при каждом шифровании — повторное использование пароля не создаёт уязвимостей
- **Верификация пароля:** AES-GCM бросает `DOMException` при несовпадении auth tag → неверный пароль детектируется без дополнительных хранимых хешей

**Почему не использовали bcrypt/scrypt из npm?** Web Crypto API — стандарт W3C, работает без зависимостей, доступен во всех современных браузерах. Сторонние крипто-библиотеки — лишняя поверхность атаки через supply chain.

### Rate limiting попыток разблокировки

Хранится в `localStorage` (`ton_wallet_attempts`):

- 0–4 неудачные попытки → немедленный ответ
- 5+ попытки → нарастающая задержка: 2^(n-5) минут, максимум 60 минут

**Зачем:** защита от перебора пароля. Да, злоумышленник с доступом к `localStorage` может сбросить счётчик — это `localStorage`-level security, а не системный. Для не-production реализации (как указано в задании) это достаточно.

### Обнуление секретных данных в памяти

JavaScript-строки иммутабельны — их нельзя безопасно занулить. Но `Uint8Array` можно:

```typescript
keys.secretKey.fill(0);  // обнуляем байты ключа перед удалением из стейта
```

При `lock()` секретный ключ обнуляется. **Компромисс:** мнемоника хранится как массив строк — их `fill('')` не гарантирует GC-очистку памяти, но явно сигнализирует о намерении. Это ограничение браузерной среды, приемлемое для не-production реализации.

### Автоблокировка

Хук `useActivityTracker` отслеживает `mousemove`, `keydown`, `click`, `touchstart` и вызывает `lock()` через 5 минут неактивности. SRP: хук не знает, что такое блокировка — он только отвечает за таймер.

## Защита от адрес-подмены

Реализована в `address-guard.ts` — пять последовательных слоёв проверки:

### Слой 1: Валидация формата

`Address.parse()` из `@ton/ton` — проверяет корректность TON-адреса (base64url, checksum). При невалидном адресе дальнейшие проверки не выполняются.

### Слой 2: Whitelist (адресная книга)

Проверяет адрес получателя против сохранённых контактов. Нормализует через `Address.parse()` — bounceable и non-bounceable форматы одного адреса считаются одинаковыми.

**Реалистичный сценарий:** пользователь отправляет средства незнакомому адресу → предупреждение `UNKNOWN_ADDRESS`. Если получатель уже есть в адресной книге — предупреждения нет.

### Слой 3: Clipboard-детектор

UI-слой (`Send.tsx`) отслеживает событие `paste` и устанавливает флаг `pastedFromClipboard`. Address Guard получает этот флаг и добавляет предупреждение `CLIPBOARD_PASTE`.

**Почему важно:** clipboard hijacking — реальная атака, при которой вредоносное ПО подменяет адрес в буфере обмена в момент вставки. Предупреждение мотивирует пользователя ещё раз сверить адрес визуально.

### Слой 4: Проверка активности адреса

Запрос к TonCenter API (`getContractState`): если адрес никогда не использовался (`state !== 'active'`), выдаётся предупреждение `INACTIVE_ADDRESS`.

**Компромисс:** слой не блокирует транзакцию при недоступности API — вместо этого добавляется мягкое предупреждение `ADDRESS_CHECK_FAILED`. Стратегия "fail open" обоснована: блокировка легитимных переводов из-за недоступности API хуже, чем пропущенное предупреждение.

### Слой 5: Порог суммы

Если сумма перевода > 50% текущего баланса — предупреждение `HIGH_AMOUNT` с уровнем `danger` (красный). Вычисляется через `BigInt` для точности без потери битов.

### Поведение предупреждений в UI

- Предупреждения отображаются на экране подтверждения **до** финальной отправки
- При наличии `danger`-предупреждений кнопка "Подтвердить" заблокирована до явного чекбокса "Я понимаю риски"
- Архитектурный принцип: предупреждения **невозможно проигнорировать случайно**

## Надёжность сети

### Circuit Breaker

`CircuitBreaker` (closed → open → half-open) защищает от каскадных ошибок:

- После 3 ошибок подряд цепь "размыкается" на 30 секунд
- В half-open пропускается один probe-запрос: успех → closed, ошибка → снова open
- Переходы состояний lazy (без таймеров) — через `resolveState()` при каждом вызове

**Почему:** без Circuit Breaker каждый запрос пользователя в период недоступности API ждёт timeout (~30с). С CB пользователь мгновенно получает ошибку с временем до следующей попытки.

### Retry с exponential backoff + full jitter

`withRetry` делает до 3 попыток с базовой задержкой 300мс:

- Задержка: `random(0, baseDelay * 2^attempt)` — full jitter предотвращает thundering herd
- Детерминированные ошибки (невалидный адрес, HTTP 429) не ретраятся

### Порядок обёрток

`TonClient SDK → withRetry → CircuitBreaker`

CB видит итоговый результат **после** всех retry. Это корректно: retry — тактические попытки восстановить конкретный запрос; CB — стратегическое решение "сервис жив?". Если все 3 retry провалились — это уже сигнал CB.

## Обработка ошибок: Result type

Все сервисные функции возвращают `Result<T, E>` (Railway-oriented programming) вместо исключений:

```typescript
type Result<T, E> = { ok: true; value: T } | { ok: false; error: E }
```

**Почему:** исключения в JavaScript не типизированы — TypeScript не заставит обработать `throw`. `Result` делает контракт явным: компилятор проверяет, что все ветки `ok/error` обработаны. Для финансового приложения это критично.

**Компромисс:** более verbose, чем `try/catch`. Принято сознательно — надёжность важнее лаконичности.

## Транзакционный flow (Send FSM)

`useSendTransaction` реализует конечный автомат:

```
idle → validating → confirming → sending → success
                                          ↘ error
```

- **idle:** форма доступна для ввода
- **validating:** запущены все проверки address-guard (async)
- **confirming:** показаны предупреждения, пользователь должен явно подтвердить
- **sending:** BOC подписан и отправлен в сеть
- **success / error:** финальный экран

**Почему FSM, а не набор boolean-флагов?** `isLoading`, `isConfirming`, `isSent` порождают невалидные комбинации состояний (например, `isLoading && isSent === true`). Дискриминированный union `{ step: 'sending' }` делает такие состояния невозможными на уровне типов.

## Ключевые компромиссы

### JS-строки и мнемоника

JS-строки иммутабельны, GC не гарантирует очистку из памяти. Мы зануляем `secretKey` (`Uint8Array.fill(0)`) сразу после подписания транзакции — но строка мнемоники, переданная из сервиса в UI, остаётся в памяти до GC. Это ограничение браузерной среды, не специфичное для этого кошелька.

### TON-мнемоника ≠ BIP39

TON использует собственный алгоритм валидации мнемоники (через `@ton/crypto`). Стандартные BIP39 слова могут не пройти `mnemonicValidate()`. В тестах используются реальные TON-мнемоники, сгенерированные `mnemonicNew()`.

### Testnet-адреса

Адреса формируются с флагом `testOnly: true`. Не переводите testnet-активы на mainnet-адреса.

### sessionStorage vs localStorage для rate limiting

Счётчик попыток пароля хранится в `localStorage`, не в `sessionStorage` — иначе перезагрузкой страницы можно обойти ограничение.

## Что добавить для production

1. **Key derivation:** обновить PBKDF2 → Argon2id (недоступен в Web Crypto, нужен WASM-полифилл) или увеличить итерации до 600k+.
2. **Аудит зависимостей:** `@ton/ton`, `@ton/crypto` — критичные пакеты, требуют регулярного `npm audit`.
3. **Secure memory:** рассмотреть `WebAssembly.Memory` с явной очисткой для хранения ключей.
4. **CSP + subresource integrity:** для production-деплоя.
5. **Hardware wallet support:** Ledger через `@ledgerhq/hw-transport-webhid`.
6. **E2E тесты:** Playwright для критичных flow (create → send → receive).
7. **Мониторинг ошибок:** Sentry с включённым scrubbing чувствительных данных.
8. **Rate limiting на уровне CB:** более агрессивные параметры для mainnet.
